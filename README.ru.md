# replytg

LLM-подсказки ответов для [telegram-business-bridge](https://github.com/AndyShaman/telegram-business-bridge):
отдельный демон, который смотрит на входящие личные сообщения (уже собранные бриджем),
генерирует два варианта ответа в твоём стиле и присылает карточку с кнопками в отдельного
Telegram-бота. Одобренный вариант уходит собеседнику **от твоего имени** через штатный
механизм черновиков бриджа.

[English README](README.md)

```
┌─ @твой_replytg_бот ─────────────┐
│ 💬 Маша: «Ты завтра сможешь?»   │
│                                 │
│ 1️⃣ Привет) да, после обеда норм │
│ 2️⃣ Завтра не обещаю, напишу     │
│                                 │
│ [1️⃣] [2️⃣] [🔄 Ещё] [✍️ Свой] [❌]│
└─────────────────────────────────┘
```

## Как это работает

- Входящее сообщение открывает «волну»: 10 минут (настраивается) входящие копятся.
- Если ты за это время ответил сам — LLM даже не вызывается.
- Иначе — одна генерация: два варианта, карточка в бота-пульт.
- Использовал подсказку (1️⃣/2️⃣/✍️) — час тишины для этого чата.
- Не использовал и не ответил — через 2 часа та же карточка придёт повторно (один раз).
- Новое сообщение от контакта — новая волна, счётчики в ноль.
- **Автоотправки нет и не будет**: каждый уходящий текст ты подтверждаешь кнопкой,
  варианты в карточке всегда показаны целиком.
- Первый запуск читает только новые сообщения — накопленную историю бриджа демон
  не трогает.
- Результат отправки приходит reply-сообщением на карточку («✅ Отправлено» / «⚠️ …»).

Интеграция с бриджем — ровно две точки:

| Что | Как |
|---|---|
| Чтение входящих/исходящих | `bridge.db` read-only (SQLite WAL) |
| Отправка одобренного ответа | вставка черновика `status='approved'` — дальше бридж сам |

Токен business-бота replytg не знает и знать не должен.

## Требования

- Работающий [telegram-business-bridge](https://github.com/AndyShaman/telegram-business-bridge)
  (версия с таблицей `drafts` и статусом `approved`; проверяется автоматически при старте).
- Python ≥ 3.12, [uv](https://docs.astral.sh/uv/).
- Любой OpenAI-совместимый LLM API (endpoint + ключ + модель).

## Установка

```bash
git clone https://github.com/consolover5-collab/replytg ~/projects/replytg
cd ~/projects/replytg       # путь важен: его ожидает systemd-юнит
uv sync
install -m 600 .env.example .env   # заполни: токен бота, свой user id, пути, LLM
cp deploy/replytg.service ~/.config/systemd/user/
systemctl --user daemon-reload
systemctl --user enable --now replytg
```

Клонируешь в другое место — поправь `WorkingDirectory`/`EnvironmentFile` в юните.

Бота-пульт создай в [@BotFather](https://t.me/BotFather) (это обычный бот, НЕ тот, что
подключён к Telegram Business). После запуска напиши ему `/start`.

При старте демон проверяет: схему bridge.db, что business-подключение ровно одно и
принадлежит `REPLYTG_OWNER_ID`. Любое несоответствие — отказ запуска с внятной ошибкой.

## Стиль-профиль

Чтобы варианты звучали как ты, а не как вежливый робот, положи в `data/style-profile.md`
описание своей манеры + примеры (шаблон: `docs/style-profile.example.md`). Два пути:

1. **Автосборка**: `uv run replytg-style` — соберёт профиль из накопленных в bridge.db
   пар «входящее → твой ответ» (нужно ≥10 пар; ответы, отправленные самим replytg,
   исключаются, чтобы модель не училась на себе).
2. **Ретроспектива**: официальный экспорт Telegram Desktop (Настройки → Продвинутые →
   Экспорт данных) и сборка профиля из него любым удобным способом.

Без профиля демон работает, но подсказки будут нейтральными.

## Приватность

- Переписка покидает машину двумя путями: запросы к выбранному тобой LLM API
  (история чата + новые входящие в промпте — выбирай провайдера осознанно) и
  карточки в твоём боте-пульте (фрагменты входящих и варианты проходят через
  серверы Telegram, как любое сообщение бота).
- Невалидные ответы LLM не логируются содержимым — только тип ошибки.
- `data/` (профиль стиля, состояние) — права 0700, защита от попадания в git встроена.
- Содержимое сообщений считается недоверенным: инструкции из переписки LLM
  проигнорирует, а отправить что-либо без твоей кнопки невозможно архитектурно.
- Доставка — best-effort: при падении процесса в узком окне между нажатием кнопки
  и вставкой черновика подтверждённый ответ может не уйти (карточка останется без
  «✅» — это сигнал проверить чат). Персистентной очереди отправки в v0.1 нет.

## Конфигурация

Все параметры — в `.env` (см. `.env.example`): окно волны, длительность тишины,
интервал повтора, лимит длины варианта, blocklist чатов, модель/endpoint LLM.
